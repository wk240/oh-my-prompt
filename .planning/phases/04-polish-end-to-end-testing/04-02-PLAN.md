# Plan 04-02: SPA Navigation Persistence Verification

## Objective

Verify and enhance MutationObserver to ensure the extension UI continues working after SPA navigation events within Lovart (e.g., switching projects, navigating between pages). The input detector should re-inject UI when new input elements appear after navigation.

## Context

Per D-12/D-13/D-14 decisions, the Content Script must continuously monitor DOM changes and automatically re-inject UI when SPA navigation occurs. Current implementation in `input-detector.ts` has basic navigation watching via a separate MutationObserver, but needs verification and potential enhancement.

**Current mechanism:**
- `InputDetector` has two observers: `observer` for general DOM changes and `navObserver` for SPA navigation
- Navigation detection compares `location.href` changes
- On navigation, `inputElement` is reset to `null` and `tryDetect()` is triggered

**Priority:** blocking

<action>
1. In `src/content/input-detector.ts`, enhance the `watchNavigation` method:

   - Add `history.pushState` and `history.replaceState` interception for more reliable SPA navigation detection:
     ```typescript
     // Intercept history API for SPA navigation
     const originalPushState = history.pushState
     const originalReplaceState = history.replaceState
     
     history.pushState = function(...args) {
       originalPushState.apply(history, args)
       this.handleNavigation()
     }
     
     history.replaceState = function(...args) {
       originalReplaceState.apply(history, args)
       this.handleNavigation()
     }
     ```

   - Add a `handleNavigation` private method that:
     - Logs `[Lovart Injector] Navigation detected via history API`
     - Resets `inputElement = null`
     - Calls `tryDetect()`

   - Also listen for `popstate` event (back/forward navigation):
     ```typescript
     window.addEventListener('popstate', () => {
       this.handleNavigation()
     })
     ```

   - Keep the existing MutationObserver-based URL watching as fallback for edge cases

2. Add cleanup for the history interception in the `stop()` method:
   - Restore original `history.pushState` and `history.replaceState`
   - Remove `popstate` listener
   - Store the original functions as private properties for restoration
</action>

### Task 3: Add Continuous MutationObserver Verification

**Priority:** medium

<read_first>
- src/content/input-detector.ts (start method and observer configuration)
</read_first>

<action>
1. In `src/content/input-detector.ts`, add a periodic health check to verify MutationObserver is still active:

   - Add a `healthCheckInterval` private property for periodic verification
   - In `start()` method, add a setInterval that runs every 30 seconds:
     ```typescript
     this.healthCheckInterval = setInterval(() => {
       if (!this.observer || !document.body.contains(this.observer.takeRecords())) {
         console.warn('[Lovart Injector] MutationObserver may be disconnected, restarting...')
         this.stop()
         this.start()
       }
     }, 30000)
     ```

   - Actually, the above approach is not reliable. Instead, use a simpler check:
     ```typescript
     this.healthCheckInterval = setInterval(() => {
       // Force a detection attempt periodically
       if (!this.inputElement) {
         this.tryDetect()
       }
     }, 30000)
     ```

   - In `stop()` method, clear the health check interval:
     ```typescript
     if (this.healthCheckInterval) {
       clearInterval(this.healthCheckInterval)
       this.healthCheckInterval = undefined
     }
     ```

2. Add the `healthCheckInterval` property declaration:
   ```typescript
   private healthCheckInterval: ReturnType<typeof setInterval> | undefined
   ```
</action>
